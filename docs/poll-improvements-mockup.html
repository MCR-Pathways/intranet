1. Custom Poll Duration
Extend the duration dropdown with two extra presets + a "Custom" date-time picker. Matches LinkedIn (presets) + StrawPoll (custom deadline).
Pros
  • Covers 95% of use cases with presets alone
  • "Custom" handles edge cases (close after a specific meeting/event)
  • Native <input type="date"> — zero dependencies, accessible
  • Minimal code change — extends existing DURATION_OPTIONS array
  • computePollClosesAt() just returns the selected ISO string for "custom"
Cons
  • Date-time picker appearance varies across browsers
  • Needs validation: can't set close date in the past
  • Timezone edge case if user is travelling

2. Export Poll Results
CSV, Excel, and PDF export via a dialog triggered from the kebab menu. Closed polls only. Poll author + system admins.
Pros
  • Three formats cover all audiences (data, managers, stakeholders)
  • Two-tier output (summary + individual) matches Typeform/Mentimeter
  • Reuses existing sanitiseCSVCell() from reports-panel
  • Kebab menu placement follows existing app convention
  • Checkbox toggles let users control exported data scope
Cons
  • PDF needs a library (jsPDF or similar) — extra bundle
  • Excel multi-sheet needs SheetJS — extra bundle
  • CSV is the only zero-dependency format
  • Individual responses reveal who voted what (privacy)
Phasing: Ship CSV first (zero dependencies). Add Excel + PDF in a follow-up PR once library options are evaluated. CSV covers the core need immediately.

3. Multi-Select Polls
Creator toggles "Allow multiple selections" at creation. Checkboxes replace radio buttons. Percentages = % of voters (Polly/Google Forms model).
Pros
  • Natural for "pick all that apply" questions
  • "% of voters" is the industry standard — intuitive
  • Simple toggle in composer — no complex UX
  • Export: semicolon-delimited choices per voter row
  • Immutable after creation — no mid-poll confusion
Cons
  • DB migration: UNIQUE constraint swap on poll_votes
  • New column on posts table (poll_allow_multiple)
  • "Change selections" UX is more complex than "Change vote"
  • Percentages exceeding 100% may confuse initially
  • Single-select enforcement moves from DB to application layer
DB change: Add poll_allow_multiple BOOLEAN DEFAULT FALSE to posts. Replace UNIQUE(post_id, user_id) on poll_votes with UNIQUE(post_id, user_id, option_id). For single-select, the server action deletes existing vote + inserts new one in a transaction.

4. Close Poll Early
Poll author and system admins can close a poll before its scheduled expiry. One server action, one menu item, no schema changes.
Pros
  • Universal feature — every enterprise platform supports this
  • Simplest of all 5 recommendations (one action, one menu item)
  • No schema change — updates existing poll_closes_at column
  • Enables the export workflow (close → export → share)
  • Prevents stale polls lingering in the feed
Cons
  • Irreversible — cannot reopen a closed poll
  • Author could close to suppress unfavourable results
  • Need to decide: should system admins close anyone's poll?

5. Increase Max Options (4 → 6)
Bump MAX_OPTIONS constant from 4 to 6. No DB change, no migration, no UI restructuring.
Pros
  • One constant change — literally MAX_OPTIONS = 6
  • No migration — DB has no option count constraint
  • Covers date-picking polls (6 consecutive weeks)
  • 6 bars still readable in the feed card
  • LinkedIn raised from 4→5 for the same reason
Cons
  • More vertical space in voting view
  • Votes spread thinner — harder to get a clear majority
  • Could encourage low-effort "dump all options" polls
